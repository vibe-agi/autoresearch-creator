# Skill Generation Templates

## Overview

This file contains templates for all files generated by `create-autoresearch`.

All generated `.md` files are **rendered from `autoresearch/pillars.json`**. Variables in templates (shown as `{{field}}` or described in prose) map to paths in pillars.json.

**CRITICAL:** No unfilled variable may remain in the generated output. Every template placeholder must be filled from pillars.json. If a value is unknown, go back to the pillar dialogue and collect it.

**CRITICAL:** Users and agents must NOT manually edit generated .md files. All changes flow through pillars.json. Manual edits will be lost on upgrade.

---

## Template 0: pillars.json

Write to: `autoresearch/pillars.json`

This is the source of truth. Schema:

```json
{
  "schema_version": "1.0",
  "generator_version": "<current autoresearch-creator version>",
  "domain": "<domain slug>",
  "created_at": "<ISO timestamp>",
  "last_updated_at": "<ISO timestamp>",
  "project_context": "<1-3 sentences describing what the project does>",
  "tier": 1,

  "frozen": {
    "primary_metric": {
      "name": "<metric name>",
      "direction": "lower-is-better | higher-is-better",
      "extraction": "<shell command or parse expression>"
    },
    "guard_metrics": [
      {
        "name": "<guard name>",
        "operator": ">=" | "<=" | "==" | "all_pass",
        "floor": "<value>",
        "command": "<command to measure or verify>"
      }
    ],
    "harness": {
      "command": "<full evaluation command>",
      "repetitions": 1
    },
    "frozen_files": [
      {"path": "<path>", "reason": "<why>"}
    ],
    "protected_patterns": [
      {
        "file": "<file path>",
        "grep_pattern": "<regex>",
        "min_count": 1,
        "protects": "<non-functional property>",
        "risk_if_removed": "<what breaks>"
      }
    ],
    "prohibited_actions": [
      "<domain-specific prohibition>"
    ]
  },

  "mutable": {
    "budget": {
      "value": 300,
      "unit": "seconds"
    },
    "surface_patterns": [
      {"pattern": "<glob>", "purpose": "<why mutable>"}
    ],
    "keep_threshold": {
      "value": 1.0,
      "unit": "percent"
    },
    "mutation_strategy_notes": [
      "<domain-specific strategy hint>"
    ],
    "meta_review_triggers": {
      "consecutive_discards": 5,
      "consecutive_crashes": 3,
      "periodic_interval": 20,
      "convergence_threshold_percent": 0.1,
      "convergence_window": 10
    }
  },

  "meta_review_history": []
}
```

### Budget Unit Enum

`mutable.budget.unit` must be one of:

- `seconds` — wall-clock time (ML training, benchmarks)
- `iterations` — step count (iterative algorithms)
- `api_calls` — external API call count (LLM experiments)
- `tokens` — LLM token count (cost-sensitive)
- `dollars` — cloud cost (resource-heavy experiments)
- `memory_mb` — memory budget
- `requests` — request count (load tests)

### Direction-Derived Operators

From `primary_metric.direction`, derive:

| direction | better_operator (is new better?) |
|-----------|---------------------------------|
| `lower-is-better` | `new < best - keep_threshold` |
| `higher-is-better` | `new > best + keep_threshold` |

`keep_threshold.unit` can be:
- `percent` — threshold is a percentage of current best (e.g., 1.0 = 1%)
- `absolute` — threshold is an absolute value

This example shows what a concrete pillars.json looks like for a Go web service optimizing memory allocations while preserving concurrency safety.

### Rendering Rules (Step-by-Step)

When rendering a template from pillars.json, follow this protocol:

1. **Load pillars.json in full.** Parse all fields.

2. **Substitute each `{{field.path}}` reference** with the literal value from pillars.json. For scalar fields, use the value directly. For lists/objects, iterate as specified by `{{#each}}` blocks.

3. **Handle optional sections**: if a field is `null`, empty array, or missing, skip the corresponding template section (for example, `mutation_strategy_notes: []` produces no bullets under "Mutation Strategy").

4. **Handle tier-conditional blocks**: some templates have variants for `tier: 1` vs `tier: 2`. Render ONLY the matching variant.

5. **Format conventions**:
   - Shell commands: wrap in backticks
   - File paths: wrap in backticks
   - Multi-item lists: use markdown bullets
   - JSON paths referenced in prose: use dotted notation like `pillars.mutable.budget.value`

6. **Validation (MUST run after rendering)**:
   - Grep the rendered file for `{{` — if any match, a substitution was missed.
   - Grep for `<...>` angle-bracket placeholders — if any match, a legacy placeholder was missed.
   - Both greps must return empty. If not, fail loudly: "Rendering incomplete — pillars.json is missing field X."

7. **Never manually "simplify" or "improve" the template during rendering.** The template is the template. Substitute values, don't paraphrase.

## Template 1: Generated SKILL.md

Write to: `.claude/skills/autoresearch-{{domain}}/SKILL.md`

````markdown
---
name: autoresearch-{{domain}}
description: Run autonomous optimization loop for {{domain}}. Iteratively improves {{primary_metric.name}} through experiment cycles.
---

# AutoResearch: {{domain}}

Run the autonomous optimization loop for this project.

## Language Rule

ALWAYS communicate with the user in their language. Detect the language from the user's messages. Technical terms (metric names, commands, file paths) stay in English, but all status updates, questions, meta-review proposals, and knowledge base entries must be in the user's language.

## Source of Truth

`autoresearch/pillars.json` is the configuration. Read it at every invocation. Never edit this or any generated file by hand — use `/create-autoresearch upgrade` to refresh templates.

## Before Starting

Read `program.md` for the complete experiment protocol.

## Quick Reference

- **Metric:** {{primary_metric.name}} ({{primary_metric.direction}})
- **Eval:** `{{harness.command}}`
- **Surface:** {{surface_patterns_summary}}
- **Budget:** {{budget.value}} {{budget.unit}} per experiment

## Version Self-Check (runs at every invocation)

```
1. Read autoresearch/pillars.json → get .generator_version
2. Compare with the current autoresearch-creator version
3. If current > saved:
   Tell user: "autoresearch-creator has upgraded from vX.Y.Z to vA.B.C.
              Run /create-autoresearch upgrade to refresh this skill."
4. Ask: upgrade now / skip this session / continue with outdated skill
5. Wait for user decision before proceeding
```

## Git Safety (runs at first experiment of each session)

```
1. Run: git status --porcelain
2. If output is non-empty:
   STOP. Show user the dirty files.
   Tell user:
     "Working tree is not clean. The autoresearch loop auto-commits and
      runs `git reset --hard`. Running on a dirty tree risks data loss.

      Options:
        1. I will commit/stash manually, then tell you to re-check
        2. Abort this run

      I will NOT auto-commit or auto-stash your changes."
3. Wait for user. Re-check after user says they're ready.
4. Only proceed when the check passes clean.
```

## Setup (first run only)

1. Verify git safety (above).
2. Check if branch `autoresearch/{{domain}}` exists:
   - If no: `git checkout -b autoresearch/{{domain}}`
   - If yes: `git checkout autoresearch/{{domain}}`
3. Verify evaluation works: run `{{harness.command}}` and extract baseline metric + guards.
4. Append baseline to `autoresearch/results.tsv`:
   ```
   <short_commit_hash>\t<baseline_metric_value>\t<guard_values>\tbaseline\tInitial baseline
   ```
5. Confirm: "Baseline {{primary_metric.name}}: <value>. Starting optimization loop."

## Pre-Experiment Sanity Check (runs BEFORE EVERY experiment)

Four cheap checks to detect codebase drift:

```
1. File existence check:
   For each file listed in pillars.frozen.frozen_files and in pillars.mutable.surface_patterns:
     - Verify it exists (or that its glob pattern matches at least one file)
   FAIL: "File X has disappeared. Code has been refactored."

2. Protected pattern integrity:
   For each entry in pillars.frozen.protected_patterns:
     count = grep -cE "{grep_pattern}" {file}
     verify: count >= {min_count}
   FAIL: "Protected pattern '{grep_pattern}' in {file} has count {actual} but expected >= {min_count}. Defensive code may have been removed."

3. Evaluation command startup smoke test:
   timeout 10 {harness.command} --help 2>&1 > /dev/null  (or `which` check)
   FAIL: "Evaluation command cannot start. Dependencies may have changed."

4. Metric extraction health check (only before the very first experiment each session):
   Run {harness.command} once, extract {primary_metric.name}
   Verify: result is finite, non-zero, within 10x of the last recorded baseline in results.tsv
   FAIL: "Metric extraction produced anomalous value. Eval tool output format may have changed."
```

If ANY check fails:
- PAUSE the loop immediately
- Report the specific failure to the user
- Offer: (1) I fixed it, re-check (2) Run /create-autoresearch upgrade (3) Abort
- Do NOT auto-recover, do NOT auto-stash, do NOT proceed under any circumstances

## The Experiment Loop

```
NEVER STOP. Once the loop begins, do NOT pause to ask the user unless:
  - Pre-experiment sanity check fails
  - Meta-review completes and proposes changes needing user approval
  - External knowledge fallback runs and no more ideas remain

If you run out of ideas, think harder — re-read the in-scope files for
new angles, try combining previous near-misses, try more radical changes.
Boredom is not a stopping condition.

loop:
  0. Pre-experiment sanity check (above) — on FAIL pause
  1. Read autoresearch/knowledge.md — check anti-patterns, reference proven patterns
  2. Analyze codebase within Surface, propose ONE improvement hypothesis
  3. Modify files (Surface only: see pillars.mutable.surface_patterns)
  4. git commit with descriptive message
  5. Run evaluation: {{harness.command}}
     If {{harness.repetitions}} > 1: run that many times, take median
  6. Extract primary metric and guard metrics
  7. Three-Layer Judgment (see program.md L1, L2, L3)
  8. Log result to autoresearch/results.tsv (ALWAYS before any git reset)
     Format: <short_commit_hash>\t<metric_value>\t<guard_values>\t<status>\t<description>
     where <status> is one of: keep | discard | crash | flag
  9. Decision based on 3-layer judgment:
     KEEP — all 3 layers pass, nothing unusual
        → status=keep, leave commit in place, advance
     FLAG (notable experiment) — see criteria below
        → status=flag
        → if L1 passed: leave commit in place (like keep)
        → if L1 failed: git reset --hard HEAD~1 (like discard)
     DISCARD — normal L1/L2/L3 failure with no notability
        → status=discard, git reset --hard HEAD~1
     CRASH — eval command errored out
        → Try 1-2 simple fixes (typo, import)
        → If still broken: status=crash, git reset --hard HEAD~1

     **When to use FLAG instead of KEEP/DISCARD:**
     - L3 returned FLAG verdict (expert was uncertain)
     - L1 strict fail but secondary metrics moved >20% (maybe primary metric is wrong)
     - Result is unexpected / not explained by existing knowledge
     - Experiment accidentally reveals a different optimization opportunity
     
     FLAG is rare by design. Meta-review treats flagged entries individually,
     not as part of the pass/fail aggregate. If >20% of experiments are
     flagged, meta-review should identify a systemic issue (usually: wrong
     primary metric).
 10. Append an entry to autoresearch/knowledge.md (flat format, see Template 4)
 11. Check meta-review triggers (see program.md)
 12. If meta-review ran and produced no actionable direction:
     → trigger external knowledge fallback (see below)
 13. Back to step 0
```

## External Knowledge Fallback

Only triggered when meta-review has run, analyzed everything, and still has no direction.

```
1. Formulate ONE targeted search query based on the current bottleneck
   Example: "FlashAttention-3 benchmark 2025 pytorch"
   Example: "Go sync.Pool alternatives high contention"
2. Use WebSearch. Hard cap: 3 searches per fallback session.
3. Prefer official docs, arXiv, GitHub issues over random blogs.
4. For each relevant finding, extract ONE actionable experiment idea.
5. Add as a candidate experiment in the next loop iteration.
6. Log the source URL in the results.tsv description field and in knowledge.md.
7. External findings do NOT bypass the 3-layer judgment. They go through the normal loop.
```

## Post-Experiment Knowledge Update

After each experiment, append ONE entry to `autoresearch/knowledge.md`:

**Format (always append, never edit existing entries):**

If KEEP:
```markdown
### PP-{{next_seq}}: {{brief title}}
- Evidence: experiment #{{N}}, commit {{hash}}
- What worked: {{description}}
- Why: {{analysis}}
- Metric delta: {{old_value}} → {{new_value}}
```

If DISCARD:
```markdown
### AP-{{next_seq}}: {{brief title}}
- Evidence: experiment #{{N}}, discarded commit {{hash}}
- What was tried: {{description}}
- Why it failed: {{analysis}}
- Metric delta: {{old_value}} → {{new_value}} (no improvement)
```

If CRASH:
```markdown
### AP-{{next_seq}}: {{brief title}} [CRASH]
- Evidence: experiment #{{N}}
- What was tried: {{description}}
- Error: {{stack trace summary}}
```

There is no "candidate" status, no promotion pipeline. The flat list IS the knowledge.
Meta-review is the only mechanism that consolidates/cleans entries.

## Meta-Review

Triggered when any of these conditions fires (defaults in pillars.mutable.meta_review_triggers):

- `consecutive_discards` consecutive discards (default: 5)
- `consecutive_crashes` consecutive crashes (default: 3)
- User manually interrupts the loop
- `periodic_interval` experiments completed (default: 20)
- Rolling avg improvement < `convergence_threshold_percent` for `convergence_window` rounds

When triggered, load `meta-review.md` and follow its protocol.
````

## Three-Layer Judgment

### L1: Metric Check (automatic, every experiment)

```
1. Extract {{primary_metric.name}} using: {{primary_metric.extraction}}

2. If harness.repetitions > 1:
     Run evaluation {{harness.repetitions}} times, take the median.

3. Compare: 
     For lower-is-better:  new < best × (1 - keep_threshold / 100)   if unit = percent
                           new < best - keep_threshold               if unit = absolute
     For higher-is-better: new > best × (1 + keep_threshold / 100)   if unit = percent
                           new > best + keep_threshold               if unit = absolute

4. Check each guard metric:
   {{#each guard_metrics}}
   - Run: `{{command}}`
   - Verify: {{operator}} {{floor}}
   {{/each}}

PASS if: primary metric improved AND all guards satisfied.
FAIL otherwise.
```

### L2: Compliance Check (automatic, every experiment)

**Part A: Mechanical protected pattern verification**

For each entry in `pillars.frozen.protected_patterns`:

```
count=$(grep -cE '{{grep_pattern}}' {{file}})
[ "$count" -ge {{min_count}} ]
```

FAIL if count is less than min_count. Any reduction below the tracked count
= automatic rejection. This catches cases where one of several occurrences
is quietly removed.

min_count is auto-detected at skill creation time by counting occurrences
in the initial codebase. Meta-review can update min_count if the agent
legitimately adds more instances of the pattern.

**Part B: Domain-specific checks**

{{#each domain_l2_checks}}
- Check: {{description}}
  Command: `{{command}}`
  Expected: {{expected}}
  FAIL if: {{failure_condition}}
{{/each}}

### L3: Secondary Review (conditional, LLM-driven)

**Mechanical trigger conditions** (any one triggers L3):

1. Primary metric improvement > 2× rolling average improvement (suspiciously good)
2. Git diff shows > 50 lines deleted total
3. Git diff matches any defensive code removal pattern (below)
4. Every 10 accumulated keeps (periodic audit, hardcoded default)

**Defensive code removal grep patterns (applied to git diff):**

```
^-.*\.Clone\(\)
^-.*sync\.(Mutex|RWMutex|Once|WaitGroup)
^-.*\b(validate|sanitize|escape|unescape)
^-.*\b(try|catch|rescue|recover|ensure)
^-.*rate[_-]?limit
^-.*(encrypt|decrypt|hash|verify|checksum)
^-.*(assert|invariant|precondition|postcondition)
^-.*\bdropout\b.*\(
```

If ANY of these patterns match the removed lines, L3 is triggered.

**L3 review prompt:**

```
You are a senior engineer reviewing a proposed change.

Change:
[git diff]

Metric change: <old> → <new>
Guard values: [...]

Review checklist:

1. Does this improvement come from real optimization or from removing a
   protection?
2. For each removed line, ask: why did this line exist?
3. Could this introduce: concurrency issues, security holes, correctness
   bugs, compliance violations?
4. Would a reasonable code reviewer merge this PR?

Common defensive patterns to watch:
- Cloning/copying shared data (concurrency)
- Locks and synchronization (thread safety)
- Input validation and sanitization (security)
- Error handling and fallbacks (fault tolerance)
- Rate limiting and circuit breakers (stability)
- Checksums and assertions (correctness)
- Regularization layers (generalization in ML)

If ANY removed line served a protective purpose, REJECT — even if the
metric improved significantly. The correct approach is to optimize the
protection in place, not remove it.

Verdict: APPROVE | REJECT <reason> | FLAG <observation for meta-review>
```

## Template 3: Generated meta-review.md

Write to: `.claude/skills/autoresearch-{{domain}}/meta-review.md`

````markdown
# Meta-Review Protocol

## Language Rule

Present all proposals and questions to the user in their language. Detect from prior conversation context. Technical terms stay in English.

## Source of Truth

This protocol modifies `autoresearch/pillars.json` only (the `mutable` section).
After changes are approved, re-render the `.md` files from the updated pillars.json.
Do NOT edit program.md, SKILL.md, or any generated .md file directly.

## When This Runs

Triggered from the main experiment loop (SKILL.md) when:

- {{meta_review_triggers.consecutive_discards}} consecutive discards
- {{meta_review_triggers.consecutive_crashes}} consecutive crashes
- User manually interrupts the loop
- {{meta_review_triggers.periodic_interval}} experiments completed
- Convergence: rolling avg improvement < {{meta_review_triggers.convergence_threshold_percent}}% for {{meta_review_triggers.convergence_window}} rounds

## Input Data

Before proposing changes, read:

- `autoresearch/pillars.json` — current configuration
- `autoresearch/results.tsv` — experiment history
- `autoresearch/knowledge.md` — accumulated anti-patterns and proven patterns

## Analysis Dimensions

### 1. Hit Rate Analysis

```
keep_rate = count(status == "keep") / count(total experiments)
```

| Rate | Interpretation | Proposed Action |
|------|---------------|----------------|
| > 40% | Strategy too conservative | Suggest more aggressive mutations, possibly architectural |
| 10-40% | Healthy range | No protocol change needed |
| < 10% | Strategy too aggressive OR direction is wrong | Suggest convergence or strategy pivot |

### 2. Diminishing Returns

Plot the magnitude of improvement for the last 10 keeps. If the trend is flat or declining:
- Approaching local optimum.
- Propose: expand `surface_patterns`, try architectural-level changes, or declare converged.

### 3. Crash Pattern Analysis

Group crashes by error type:
- If same error appears 3+ times: add a specific entry to `mutation_strategy_notes` warning against that pattern.
- If crashes cluster in a specific file: consider removing that file from `surface_patterns`.

### 4. Mutation Direction Retrospective

Categorize experiments by change type (e.g., hyperparameter, architecture, algorithm, config). Identify:
- Categories with highest keep rate → reinforce in `mutation_strategy_notes`.
- Categories with 0% keep rate over 5+ attempts → de-prioritize in `mutation_strategy_notes`.

### 5. Flagged Experiment Review (CRITICAL — read individually)

Flagged entries (`status=flag` in results.tsv) are NOT part of the aggregate
pass/fail rate. They are experiments the agent singled out as notable.
Read each one and decide:

- **Pattern across multiple flags**: e.g., "3 flagged experiments all showed
  bytes_per_op dropping 90%+ while allocs_per_op stayed flat" → propose
  changing the primary metric to the one that actually moves
- **Single flag with major insight**: propose surface expansion, budget
  adjustment, or new strategy notes based on the observation
- **L3 FLAG-verdict cluster**: review the expert observations, extract
  common themes, update mutation_strategy_notes

**Design smell alert**: if >20% of experiments are flagged, the primary
metric probably doesn't capture what the user actually cares about.
Recommend the user re-run `/create-autoresearch` with a different metric.

### 6. Knowledge Consolidation

Review `autoresearch/knowledge.md`:
- Merge duplicate entries (same anti-pattern observed in multiple experiments).
- Mark superseded entries with reason (don't delete — archive).
- Extract cross-cutting patterns (e.g., "3 different attention tweaks all failed the same way" → general rule).

## Output: Proposed Modifications to pillars.json

Present proposed changes to the user **one at a time**:

```
Meta-Review Proposal [N/M]:

Change: {json path e.g. pillars.mutable.keep_threshold.value}
        {old value} → {new value}

Rationale: {why, based on which analysis dimension}

Evidence:
- {experiment numbers or observations supporting this change}

Approve / Reject / Modify?
```

## Safety Boundary

Meta-review **CANNOT** propose changes to `pillars.frozen.*`:

- primary_metric (name, direction, extraction)
- guard_metrics (names, operators, floors, commands)
- harness (command, repetitions)
- frozen_files
- protected_patterns
- prohibited_actions

If meta-review concludes that a frozen field is fundamentally wrong (e.g., "the chosen metric doesn't actually reflect the project goal"), it may ONLY report this to the user:

```
FROZEN RULE CONCERN:

I believe {frozen field} may need reconsideration because {reason}.

This requires re-running `/create-autoresearch` from scratch to redefine
the optimization problem. I cannot modify frozen rules within the loop.
```

## Application

After user approves each change:

1. Apply the change to `autoresearch/pillars.json`:
   - Modify the target field under `mutable.*`
   - Append an entry to `meta_review_history` with:
     - timestamp
     - the affected field path
     - old and new value
     - rationale
     - supporting evidence
   - Update `last_updated_at`

2. Re-render the generated .md files from the updated pillars.json.

3. Update `autoresearch/knowledge.md` with any consolidation actions.

4. Stage only the affected paths (note: knowledge.md is gitignored, don't stage it):
   ```
   git add autoresearch/pillars.json \
           .claude/skills/autoresearch-{{domain}}/SKILL.md \
           .claude/skills/autoresearch-{{domain}}/program.md \
           .claude/skills/autoresearch-{{domain}}/meta-review.md
   git commit -m "autoresearch: meta-review #<N> — <short summary>"
   ```
   
   knowledge.md changes are NOT committed — they persist in the working
   tree but are intentionally excluded from git because they accumulate
   across git-reset boundaries (see .gitignore).

5. Resume the experiment loop.
````

## Template 5: results.tsv

Write to: `autoresearch/results.tsv`

**5 columns** (karpathy-compatible format):

```tsv
commit	primary_metric	guard_metrics	status	description
```

Fields:
- `commit` — git short hash (7 chars)
- `primary_metric` — the scalar value (as string, original precision)
- `guard_metrics` — semicolon-separated `name:value` pairs, e.g., `vram:12.1;race:pass`
- `status` — one of: `baseline` | `keep` | `discard` | `crash` | `flag`
- `description` — brief description; may include external knowledge source URL

**Status semantics:**

| Status | Code state after logging | When to use |
|--------|--------------------------|-------------|
| `baseline` | unchanged | first entry, establishes the anchor |
| `keep` | commit stays | all 3 layers passed, normal win |
| `discard` | `git reset --hard HEAD~1` | L1 or L2 or L3 failed, nothing notable |
| `crash` | `git reset --hard HEAD~1` | eval command errored out, not a metric comparison |
| `flag` | depends on L1: stays if L1 passed, resets if L1 failed | experiment is notable beyond normal aggregation and needs meta-review attention (see below) |

**When to mark an experiment as `flag`:**

Use `flag` instead of `keep`/`discard` when ANY of these holds:
- L3 returned `FLAG` verdict (expert was uncertain but didn't reject)
- L1 strict fail but guard/secondary metrics showed >20% improvement (maybe the primary metric is wrong)
- Primary metric improved but L2 raised an observation worth investigating
- Result is unexpected given the existing knowledge base
- An experiment accidentally reveals a different optimization opportunity

The agent chooses `flag` deliberately — it's not an automatic verdict. The
agent is saying "this is worth reading later, don't just count it in the
aggregate stats". Meta-review treats flagged entries individually, not as
part of the pass/fail rate.

**Important:** `flag` is rare by design. If more than ~20% of entries are
flagged, the meta-review triggers should identify a systemic issue (usually:
the primary metric is wrong).

Initialize with **just the header row** (no data rows). The loop appends rows during execution.

**CRITICAL:** Always log to results.tsv BEFORE any `git reset --hard`. If the commit hash is reset first, it cannot be recorded afterward.

**CRITICAL: results.tsv MUST be gitignored.** It is an append-only log that must survive `git reset --hard` on discarded experiments. If committed, the reset would erase log entries, losing the record of discarded attempts. See Phase 3 Step 5 in the factory SKILL.md for the .gitignore setup.

## Rendering Notes

When rendering templates from pillars.json:

1. **Handlebars-style syntax** (`{{field}}`, `{{#each}}`, `{{#if}}`) is a notation for readability in this file. The actual rendering is performed by the LLM reading pillars.json and substituting values — not by a templating engine.

2. **If a field in pillars.json is missing/null**, use the defaults described in the schema or leave the template section out (for optional sections like `domain_common_directions`).

3. **Tier 1 vs Tier 2**: the generated SKILL.md renders ONE loop variant based on `pillars.tier`. Tier 1 uses `repetitions = 1`. Tier 2 uses whatever `harness.repetitions` specifies (usually 3-5).

4. **Tier 3 is not supported in v0.5.0.** If pillar dialogue detects Tier 3, report to user that interactive loops are not supported and suggest a proxy metric that can be evaluated locally.

5. **After rendering**, validate: no unfilled `{{...}}` placeholders remain. If any remain, it means a pillars.json field is missing — fail loudly rather than silently produce an invalid skill.
